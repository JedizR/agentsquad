# Architecture

How AgentSquad is structured, how control flows, and where files live.

---

## System overview

```
┌─────────────────────────────────────────────────────────────────┐
│  Your machine                                                   │
│                                                                 │
│  You ──► Claude Code (Tech Lead)                                │
│               │                                                 │
│               ├──► make backend  ──► Gemini Flash ──┐           │
│               ├──► make frontend ──► Gemini Flash ──┤           │
│               ├──► make qa       ──► Gemini Lite  ──┤──► agent-output/
│               ├──► make devops   ──► Gemini Lite  ──┤           │
│               └──► make consult  ──► Gemini Pro   ──┘           │
│                        │                                        │
│                        └──► Google AI API (single key)          │
└─────────────────────────────────────────────────────────────────┘
```

**Claude Code** holds the architecture context, makes decisions, and reviews output. It never runs code — it delegates.

**Gemini agents** execute tasks in parallel. Each one is a stateless CLI invocation with a SKILL.md loaded automatically.

---

## Control flow

A typical feature delivery cycle:

```
1. You describe the feature to Claude
2. Claude writes api-contracts/<feature>.md
3. Claude dispatches agents with make targets
4. Gemini agents write to agent-output/
5. Claude reads output, resolves [ESCALATE TO CLAUDE] flags
6. Claude integrates the reviewed code into the project
7. Claude commits
```

No agent has write access to `src/` or any production path. All output goes to `agent-output/` (gitignored). Integration is always a human + Claude step.

## Filesystem layout

### Global home (`~/.agent-team/`)

Portable — runs on any machine after `setup.sh`.

```
~/.agent-team/
├── .env.team                    # API keys (chmod 600, never committed)
├── .env.team.example            # Template (safe to share)
├── .gitignore                   # Excludes .env.team
├── scripts/
│   ├── setup.sh                 # One-shot setup (idempotent)
│   ├── health-check.sh          # Verify all 5 agents respond
│   ├── gemini-call.sh           # Retry wrapper + escalation detector
│   ├── strip-filler.sh          # Strip LLM preamble from output
│   ├── rotate-key.sh            # Swap to reserve key on 429
│   └── migrate.sh               # rsync to new machine
└── skills/
    ├── backend-engineer/
    │   ├── SKILL.md
    │   └── references/
    │       ├── api-patterns.md
    │       ├── db-patterns.md
    │       └── output-format.md
    ├── frontend-engineer/       (same structure)
    ├── qa-engineer/             (same structure)
    ├── devops-engineer/         (same structure)
    ├── business-consultant/     (same structure)
    └── tech-lead/               (same structure)
```

### Gemini skill symlinks (`~/.gemini/skills/`)

`setup.sh` creates symlinks here pointing to `~/.agent-team/skills/`. Gemini CLI reads this directory automatically on startup.

```
~/.gemini/skills/
├── backend-engineer   -> ~/.agent-team/skills/backend-engineer/
├── frontend-engineer  -> ~/.agent-team/skills/frontend-engineer/
├── qa-engineer        -> ~/.agent-team/skills/qa-engineer/
├── devops-engineer    -> ~/.agent-team/skills/devops-engineer/
└── business-consultant -> ~/.agent-team/skills/business-consultant/
```

### Project directory (per-project)

Generated by `/init-agent-team` inside Claude Code:

```
your-project/
├── CLAUDE.md              # Claude's session context + team config
├── TEAM.md                # Agent roster + rules for this project
├── Makefile               # make backend / make qa / make team-health
├── .gemini/
│   └── settings.json      # Enables Google Search for Consultant
├── api-contracts/         # Claude writes these; agents read them
│   └── auth.md
└── agent-output/          # gitignored; all Gemini output lands here
    ├── .gitkeep
    ├── backend-auth-1741234567.ts
    ├── frontend-auth-1741234567.tsx
    └── qa-auth-tests-1741234567.ts
```

## SKILL.md activation

Gemini CLI uses semantic similarity between the task prompt and the `description` field in each SKILL.md frontmatter to decide which skill to load.

```yaml
---
name: backend-engineer
version: 1.0.0
description: Activate when implementing server-side code including REST or
             GraphQL API endpoints, database schemas, authentication logic,
             or any Node.js/TypeScript backend service.
---
```

When you run `make backend TASK="implement the users API"`, Gemini scores the task against all skill descriptions in `~/.gemini/skills/` and activates the best match. This is why the task wording matters — see [Troubleshooting](troubleshooting.md#5-skillmd-not-activating).

## Credential flow

```
~/.agent-team/.env.team
    └── GEMINI_API_KEY=AIzaSy...
        └── source'd into shell via ~/.zshrc (or manually)
            └── Makefile passes as: GEMINI_API_KEY=$GEMINI_API_KEY gemini ...
                └── gemini-call.sh reads it as: GEMINI_API_KEY="$key"
```

Multi-key setup (optional): set `GEMINI_KEY_BACKEND`, `GEMINI_KEY_FRONTEND`, etc. in `.env.team`. The Makefile uses `$(or $(GEMINI_KEY_BACKEND),$(GEMINI_API_KEY))` — per-agent key if set, shared key otherwise.

## Output pipeline

Raw Gemini output passes through `strip-filler.sh` before being written to `agent-output/`:

```
Gemini CLI stdout
  └── gemini-call.sh captures it
      └── writes to output_file.raw
          └── strip_filler() processes it
              ├── Strategy 1: if code blocks found → extract code only
              └── Strategy 2: strip conversational preamble patterns
                  └── writes to output_file (final)
                      └── grep "[ESCALATE TO CLAUDE]" — alert if found
```

---

## Retry and failover

`gemini-call.sh` wraps every Gemini call with:

1. **3 retry attempts** with exponential backoff (15s, 30s, 45s)
2. **429 detection** — switches to `GEMINI_KEY_RESERVE_1` on the 2nd attempt if set
3. **401/403 detection** — exits immediately (bad key, no point retrying)
4. **500/503 detection** — retries with shorter backoff (5s, 10s, 15s)

## Relationship to agentskills.io

SKILL.md files follow the [agentskills.io](https://agentskills.io) open format. The format is model-agnostic — the same SKILL.md works with any CLI that supports skill loading from a directory. Currently only Gemini CLI is wired up in AgentSquad, but the format is ready for OpenAI or Ollama integrations (contributions welcome).
